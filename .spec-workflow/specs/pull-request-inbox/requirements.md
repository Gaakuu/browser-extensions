# 要件定義書

## 概要

Chrome 拡張機能「Pull Request Inbox」は、GitHub の Pull Request をサイドパネル UI で一元管理するツール。GitHub Personal Access Token (PAT) を使ってユーザーに関係する PR（自分がレビュワー／自分が作成者）を定期的に取得し、役割（Reviewer / Author）×リポジトリで階層的にリッチ表示する。サイドパネルに並ぶ PR カードをクリックすれば、既存の PR タブにフォーカスするか、新しいタブで開く。ユーザーが開いている PR タブの favicon は CI 状態・レビュー状態に応じて動的に書き換わり、タブを見るだけで進行状況がわかる。要対応の PR 件数は拡張機能ツールバーアイコンのバッジに表示され、作業中でも気づける。タブを勝手に開いたり閉じたりはしない — サイドパネルが「受信箱」として機能し、ユーザーが能動的に対応する設計。

## 要件

### 要件 1: PAT による GitHub 認証と安全な保管

**ユーザーストーリー:** ユーザーとして、自分の GitHub Personal Access Token を拡張機能に登録して、プライベートリポジトリを含む自分宛の PR を取得したい。なぜなら、業務で扱う PR の大半がプライベートリポジトリにあるから。また、PAT は機密情報なので可能な限り安全に保管されてほしい。

#### 受け入れ基準

1. 拡張機能のオプションページで PAT を入力・保存できること
2. PAT 入力欄は `type="password"` + `autocomplete="off"` + `spellcheck="false"` を指定し、画面録画・ブラウザパスワードマネージャ・IME 変換候補への露出を最小化すること
3. PAT はデフォルトで `chrome.storage.session`（メモリのみ、ブラウザセッション終了で消失）に保存されること。Chrome 公式が「機密データは `storage.local`/`sync` ではなく `storage.session` に」と推奨しているため、これを第一選択とする
4. `chrome.storage.session.setAccessLevel('TRUSTED_CONTEXTS')` で content script からのアクセスを完全に遮断すること
5. オプションページで「このブラウザに記憶する（永続化）」トグルを提供し、ON の場合のみ PAT を `chrome.storage.local` に Web Crypto API (AES-GCM) で暗号化して保存すること。OFF がデフォルト
6. 暗号化鍵はデバイス固有の情報＋拡張機能固有のソルトから PBKDF2 で導出し、鍵自体は保存しないこと（多層防御として位置づけ、完全な秘匿性は保証しない旨をオプションページに注意書きで明示）
7. `storage.sync` は絶対に使用しない（PAT が Google アカウント経由で同期されるリスクを避ける）
8. **Fine-grained PAT を推奨**: オプションページに GitHub の Fine-grained PAT 発行ページへのリンクと、必要最小限のスコープ（対象リポジトリ限定 + `Pull requests: Read-only`, `Metadata: Read-only`）の明示的なガイドを表示すること
9. **有効期限の推奨**: PAT 発行ガイドで「30〜90 日の有効期限設定」を推奨し、期限切れが近い場合はサイドパネルに警告を表示すること
10. 保存時に GitHub API `/user` エンドポイントで PAT の有効性と対応スコープを検証し、成功時はユーザー名を表示、失敗時はエラーを表示すること
11. PAT 未設定時はサイドパネルに「PAT を設定してください」という誘導と設定画面へのリンクを表示すること
12. PAT を削除する操作を提供し、削除時は session / local の双方をクリアし、関連するキャッシュもリセットすること

### 要件 2: PR の自動取得（ポーリング）

**ユーザーストーリー:** ユーザーとして、自分宛の PR が新しく来たら自動で検知してほしい。なぜなら、GitHub を定期的に確認しに行く手間を省きたいから。

#### 受け入れ基準

1. Background Service Worker が `chrome.alarms` API で定期ポーリング（デフォルト: 2分間隔、設定可能範囲: 1〜10分）を行うこと
2. ポーリングでは GitHub Search API を 2 回呼び出し、以下を取得すること
   - `is:open is:pr review-requested:@me` → レビュワーとしての PR
   - `is:open is:pr author:@me` → 作成者としての PR
3. 取得した PR は内部ストア（`chrome.storage.local`、PAT とは別領域、機密情報は含まない）にキャッシュされ、前回取得時との差分から「新着 PR」「更新された PR」「クローズ/マージされた PR」「レビュー完了した PR」を判定できること
4. GitHub API のレート制限（X-RateLimit-Remaining）を監視し、残量が 100 未満になった場合はポーリング間隔を一時的に延長すること
5. ネットワークエラー時は指数バックオフでリトライし、3 回連続失敗したらサイドパネルにエラーを表示すること
6. マージ・クローズされた PR、および自分のレビューが完了した PR（Approve または Request changes を送信済み）は、次回ポーリング時にサイドパネルから自動的に除去されること（**タブ操作は一切行わない**）
7. ユーザーが「レビュー完了後もサイドパネルに残す」設定（要件 6）を ON にしている場合、レビュー完了を理由にした除去はスキップし、マージ/クローズまで残すこと

### 要件 3: サイドパネル UI での PR 一覧表示とタブ切り替え

**ユーザーストーリー:** ユーザーとして、今自分が対応すべき PR の一覧を常に横に表示しておきたい。なぜなら、作業しながら PR の状況を俯瞰し、必要なものにすぐ切り替えたいから。

#### 受け入れ基準

1. 拡張機能のツールバーアイコンをクリックすると `chrome.sidePanel` が開くこと
2. サイドパネルには「Reviewer」「Author」の 2 セクションが表示され、それぞれリポジトリ（`owner/repo`）ごとにサブグルーピングされた折りたたみ可能なリッチカードリストが見えること
3. 各 PR カードには次の情報を表示すること: リポジトリ名、PR 番号、タイトル、作成者アバター、ステータスアイコン、最終更新時刻（相対表示）、CI 進捗サマリ（成功/失敗/実行中の数）、レビュー進捗サマリ（必要数のうち Approve 済み件数）
4. カードをクリックしたとき、対応する PR タブが**既に存在する**場合: そのタブに `chrome.tabs.update({active: true})` + `chrome.windows.update({focused: true})` でフォーカスすること
5. カードをクリックしたとき、対応する PR タブが**存在しない**場合: `chrome.tabs.create({ url, active: true })` で新規タブを開くこと（タブグループ化は行わない）
6. リストは最終更新日時降順でソートされ、「新着」PR にはバッジを表示すること
7. 手動ポーリングをトリガーする「更新」ボタンが、サイドパネルのヘッダ部にアイコンボタン（`refresh` アイコン）として**さりげなく**配置されていること（リストを圧迫しない最小サイズ、ホバーでツールチップ表示）。実行中はアイコンが回転するなどのフィードバックを示し、完了時に最終更新時刻を相対表示で近くに表示すること
8. 各リポジトリセクションは折りたたみ可能で、折りたたみ状態は永続化されること（`chrome.storage.local` の UI 状態領域）
9. サイドパネルはキーボードでも操作可能であること（↑↓ でフォーカス移動、Enter で該当タブへ）

### 要件 4: favicon による PR ステータスの可視化（ユーザーが開いたタブにのみ適用）

**ユーザーストーリー:** ユーザーとして、自分が開いている PR タブの favicon を見ただけで PR の CI・レビュー状況がわかるようにしたい。なぜなら、タブを開き直さずにどの PR が action required かを把握したいから。

#### 受け入れ基準

1. ユーザーが PR タブ（`https://github.com/*/*/pull/*`）を開いているときのみ、その Content Script が Background から受け取ったステータス情報に基づいて `<link rel="icon">` を動的に書き換えること（拡張機能が能動的にタブを生成することはない）
2. ステータスは以下の優先度で 1 つが決定されること（上ほど優先）
   - `ci_failing` (赤): CI チェックに失敗がある
   - `changes_requested` (オレンジ): レビューで変更要求がある
   - `approved` (緑): 必要数の Approve を満たしている
   - `ci_pending` (黄): CI が進行中
   - `review_required` (青): レビュー待ち（デフォルト）
3. 各ステータスに対応する SVG/PNG の favicon が拡張機能同梱で用意されていること
4. ステータスが変化したとき（次回ポーリング時）、既に開いている PR タブの favicon も更新されること
5. Content Script は MutationObserver でページ側からの favicon 再設定に対しても、ステータス favicon を維持すること
6. マージ・クローズ・レビュー完了した PR についてはステータス favicon を解除し、GitHub の元の favicon に戻すこと

### 要件 5: ツールバーアイコンのバッジ表示

**ユーザーストーリー:** ユーザーとして、サイドパネルを開いていないときでも、対応すべき PR が溜まっていることに気づきたい。なぜなら、作業に集中している間も重要な変更を見逃したくないから。

#### 受け入れ基準

1. 拡張機能のツールバーアイコンに、未対応 PR の件数をバッジとして表示すること（`chrome.action.setBadgeText`）
2. 「未対応 PR」の定義: 次のいずれかに該当する PR
   - 自分がレビュワーで、まだレビューを送っていない（`review_required`, `ci_failing` 等、要対応ステータス）
   - 自分が作成者で、CI 失敗または変更要求を受けている
3. バッジの背景色はステータスに応じて変化すること
   - 赤 (`#d73a49`): `ci_failing` または `changes_requested` を含む場合
   - 黄 (`#dbab09`): `ci_pending` のみ
   - 青 (`#0366d6`): `review_required` のみ
   - バッジなし: 未対応 0 件
4. バッジの数値が 99 を超える場合は `99+` と表示すること
5. ポーリングで PR リストが更新されたタイミングで、バッジがリアルタイムに更新されること

### 要件 6: 設定画面

**ユーザーストーリー:** ユーザーとして、PAT の登録・ポーリング間隔の調整・サイドパネルの表示方針を細かく設定したい。なぜなら、自分のワークフローに合わせて動作をチューニングしたいから。特に、自分がレビュワーの PR については、レビュー完了後もマージ状況を追いたい場合があるので、その挙動を選びたい。

#### 受け入れ基準

1. オプションページ（`options.html`）に以下の設定項目があること
   - **PAT**: 入力欄（マスク表示、検証ボタン）
   - **PAT の永続化**: 「このブラウザに記憶する（暗号化して保存）」トグル（デフォルト OFF）
   - **ポーリング間隔**: 1〜10 分のスライダーまたはセレクト（デフォルト 2 分）
   - **レビュー完了後もサイドパネルに残す**: ON/OFF トグル（デフォルト OFF。ON にすると自分のレビュー完了を理由にした除去をスキップし、マージ/クローズまで残す）
   - **バッジの表示**: ON/OFF トグル（デフォルト ON）
2. PAT 以外の設定は `chrome.storage.local` に保存され、Background・サイドパネル・Content Script で共通の設定を参照すること
3. 設定変更は即座に反映されること（次回ポーリングから新しい間隔、トグルは即座に効く）
4. 「永続化」トグルを OFF→ON に切り替えた場合、その場で暗号化して local に保存すること。ON→OFF に戻した場合は local から削除し session のみに残すこと

## 非機能要件

### コードアーキテクチャ
- **単一責任原則**: 各ファイルは明確な1つの責務を持つ
- **モジュラー設計**: API クライアント、ポーリング、ストア、UI を分離し再利用可能にする
- **明確なインターフェース**: Background ↔ Content Script ↔ Side Panel 間のメッセージ型を厳密に定義する
- **既存パターンの踏襲**: `extensions/tab-switcher/` のエントリポイント構成、メッセージングパターン、`packages/shared/` のストレージユーティリティ、`packages/ui/` のテーマに従う

### パフォーマンス
- サイドパネル起動から PR リスト描画まで 300ms 以内（キャッシュ表示）
- PR ポーリングの API コール時間が 5 秒を超える場合はタイムアウトする
- favicon の書き換えは 1 タブあたり 50ms 以内に完了すること
- キャッシュされた PR データが 1000 件を超えた場合、除去後 24 時間経過したものから削除する

### セキュリティ

#### 権限の最小化
- Chrome 権限は最小限に絞る: `storage`, `tabs`, `alarms`, `sidePanel`, `action`, `host_permissions: https://api.github.com/*`, `https://github.com/*/pull/*`
- `tabGroups` 権限は使用しない（タブグループ機能を持たないため）
- `<all_urls>` や `activeTab` を要求しない

#### PAT 保管
- PAT は原則 `chrome.storage.session` のみに保存し、`setAccessLevel('TRUSTED_CONTEXTS')` で content script から遮断する
- 永続化が有効な場合でも `chrome.storage.local` には Web Crypto API (AES-GCM) で暗号化したバイト列のみを保存し、平文を残さない
- `chrome.storage.sync` は PAT の保存先として使用しない
- GitHub API リクエスト時以外に PAT をメモリ外へ送信しない
- Content Script から Background へ PAT を渡すメッセージを作らない（Content Script は PAT を一切扱わない）
- オプションページには「完全な秘匿性は保証されない（拡張機能のコードを解析すれば鍵導出手順は再現可能）」旨の注意書きを明示する

#### Web ページ・他拡張機能からの遮断
- `manifest.json` に `externally_connectable: {}` を明示的に指定し、外部 Web ページや他拡張機能からのメッセージパッシングを完全に禁止すること
- `web_accessible_resources` は必要最小限（favicon 画像ファイルのみ）に限定し、対象 URL も `matches: ["https://github.com/*"]` で絞ること
- Content Script は ISOLATED world で動作させ、ホストページの JS と名前空間を共有しないこと（WXT / MV3 のデフォルト動作）

#### CSP (Content Security Policy)
- `manifest.json` の `content_security_policy` を厳格に設定すること
  - `extension_pages`: `script-src 'self'; object-src 'self'; connect-src 'self' https://api.github.com https://github.com`
  - `'unsafe-eval'` / `'unsafe-inline'` / 任意の CDN は許可しない
- `connect-src` で GitHub 以外への外部通信を禁止し、万が一の XSS でも PAT を外部へ exfiltrate できないようにする

#### 被害範囲の限定
- Fine-grained PAT の使用をガイドで強く推奨し、対象リポジトリとスコープを最小化することで、万が一漏洩した場合でも被害を局所化する
- PAT の有効期限設定（30〜90 日）をガイドで推奨し、期限切れを検知したらサイドパネルに警告を表示する

### ユーザビリティ
- PAT 未設定時のガイド表示がわかりやすく、オプションページへの導線があること
- **勝手にタブを開いたり閉じたりしない**: ユーザーの現在の作業を一切邪魔しない。PR タブは必ずユーザーが能動的にサイドパネルのカードをクリックしたときだけ開かれる
- ダークモード / ライトモードの両方に対応すること（`packages/ui/` のテーマを利用）
- サイドパネルはキーボードでも操作可能であること（↑↓ でフォーカス移動、Enter で該当タブへ）

## スコープ外

- GitHub OAuth / GitHub App 認証（v1 は PAT のみ）
- GitHub Enterprise Server (self-hosted) の対応（v1 は github.com のみ）
- Issue・Discussion の管理（PR のみ）
- PR コメント・レビューコメントの拡張機能内での投稿（タブで開いて GitHub 上で行う）
- PR のマージ操作
- デスクトップ通知（OS の通知センター連携）
- 複数アカウント対応
- OS のキーチェーン連携（Native Messaging 経由）
- **PR タブの自動オープン／自動クローズ／タブグループ管理**（ユーザーの作業を邪魔しない設計思想のため）
